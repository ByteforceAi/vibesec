# 취약점 탐지 규칙 (Detection Rules)

> 각 취약점을 어떻게 찾아내는지 정리한 내부 참조 문서입니다.
> 기술적 세부사항이 포함되어 있습니다 (내부용).

---

## SECRETS 카테고리

### VULN-001 — AI 서비스 비밀번호가 화면 코드에 그대로 보여요

**탐지 방법**
1. 프론트엔드 번들(dist/, build/, .next/, .svelte-kit/) 내 .js 파일 검색
2. 패턴: `sk-[a-zA-Z0-9]{20,}` (OpenAI 키)
3. 패턴: `sk-proj-[a-zA-Z0-9_-]+` (OpenAI 프로젝트 키)
4. 소스맵(.map) 파일에서도 동일 패턴 검색

**False positive 처리**
- `sk-test-` 또는 `sk-xxx` 같은 더미 키는 경고만
- 주석 안에 있는 키도 주의 등급으로 표시

**자동화 가능?**
예, grep/ripgrep으로 즉시 가능.

---

### VULN-002 — 데이터베이스 관리자 비밀번호가 코드에 들어있어요

**탐지 방법**
1. 코드에서 `supabase` + `service_role` 키워드 조합 검색
2. 패턴: `eyJhbGciOi` (Base64 JWT 시작) 뒤에 긴 문자열
3. 환경변수명: `SUPABASE_SERVICE_ROLE_KEY`, `SUPABASE_SERVICE_KEY`

**False positive 처리**
- `anon` 키는 프론트엔드에서 정상 사용 가능 (하지만 RLS가 꺼져있으면 여전히 위험)

**자동화 가능?**
예, 정규식 검색으로 가능.

---

### VULN-003 — 비밀번호 파일이 누구나 볼 수 있는 곳에 올라갔어요

**탐지 방법**
1. 깃허브 레포 루트에서 `.env`, `.env.local`, `.env.production`, `.env.development` 존재 확인
2. `.gitignore`에 `.env*` 패턴이 없는 경우 경고
3. git log에서 `.env` 파일의 add/commit 이력 검색

**False positive 처리**
- `.env.example` (값이 비어있는 템플릿)은 정상

**자동화 가능?**
예, GitHub API 또는 git ls-files로 확인 가능.

---

### VULN-004 — 결제 시스템 비밀번호가 노출돼 있어요

**탐지 방법**
1. 패턴: `sk_live_[a-zA-Z0-9]{24,}` (Stripe Secret Key)
2. 패턴: `rk_live_[a-zA-Z0-9]{24,}` (Stripe Restricted Key)
3. 프론트엔드 번들과 서버 코드 모두 검색

**False positive 처리**
- `pk_live_` (Publishable Key)는 프론트엔드에서 정상 사용
- `sk_test_`는 테스트 키이므로 주의 등급

**자동화 가능?**
예, 정규식 검색.

---

### VULN-005 — 비밀번호가 코드 안에 직접 적혀있어요

**탐지 방법**
1. 패턴: `password\s*[:=]\s*["'][^"']+["']`
2. 패턴: `apiKey\s*[:=]\s*["'][^"']+["']`
3. 패턴: `secret\s*[:=]\s*["'][^"']+["']`
4. 환경변수가 아닌 하드코딩된 문자열 할당 탐지

**False positive 처리**
- 환경변수 참조(`process.env.X`)는 정상
- 빈 문자열 할당은 무시

**자동화 가능?**
예, AST 파서 또는 정규식으로 가능.

---

### VULN-006 — 비밀번호를 지웠지만 과거 기록에 남아있어요

**탐지 방법**
1. `git log --all -p | grep -E "(sk-|SUPABASE_|password|secret)"`
2. `git log --all --diff-filter=D -- "*.env*"`
3. trufflehog, gitleaks 같은 시크릿 스캐너 실행

**False positive 처리**
- 테스트/더미 값은 주의 등급으로 분류

**자동화 가능?**
예, gitleaks/trufflehog 자동화 가능.

---

### VULN-007 — 비밀번호를 카톡이나 슬랙으로 공유했어요

**탐지 방법**
1. 수동 확인 필요 — 채팅 히스토리 검색
2. 슬랙의 경우 검색 API로 키 패턴 검색 가능

**자동화 가능?**
부분적. 슬랙 API로 일부 자동화 가능하나 카톡은 수동.

---

### VULN-008 — 비밀번호를 한 번도 바꾼 적 없어요

**탐지 방법**
1. 각 서비스 대시보드에서 키 생성일 확인
2. 90일 이상 된 키는 경고

**자동화 가능?**
부분적. API가 있는 서비스만 자동화 가능.

---

### VULN-009 — 테스트용 비밀번호와 진짜 비밀번호가 섞여있어요

**탐지 방법**
1. `.env` 파일에서 test/prod 키가 혼재되어 있는지 확인
2. `_test_`와 `_live_` 키가 같은 파일에 있는지 검색

**자동화 가능?**
예, 패턴 매칭으로 가능.

---

### VULN-058 — Anthropic(Claude) 비밀번호가 코드에 보여요

**탐지 방법**
1. 패턴: `sk-ant-[a-zA-Z0-9_-]{20,}`
2. 프론트엔드 번들 및 서버 코드에서 검색

**자동화 가능?**
예, 정규식.

---

### VULN-059 — 클라우드 서버 비밀번호가 코드에 있어요

**탐지 방법**
1. 패턴: `AKIA[0-9A-Z]{16}` (AWS Access Key ID)
2. 패턴: AWS Secret Key는 40자 Base64 문자열
3. gitleaks, trufflehog로 종합 스캔

**자동화 가능?**
예, 전용 스캐너 사용 권장.

---

### VULN-060 — 비밀번호 파일을 보호 목록에 안 넣었어요

**탐지 방법**
1. `.gitignore` 파일에 `.env*` 라인이 있는지 확인
2. 없으면 즉시 경고

**자동화 가능?**
예, 단순 파일 읽기.

## AUTH 카테고리

### VULN-010 — 로그인 안 해도 개인 페이지에 들어갈 수 있어요

**탐지 방법**
1. 보호 대상 API 라우트에 인증 미들웨어/가드 존재 여부 확인
2. curl로 토큰 없이 보호 API 호출 시 200 응답 여부 확인
3. Next.js: `middleware.ts` 존재 및 라우트 보호 확인
4. SvelteKit: `hooks.server.ts`에서 인증 체크 확인

**자동화 가능?**
예, 무인증 API 호출 테스트 자동화 가능.

---

### VULN-011 — 관리자 페이지에 아무나 들어갈 수 있어요

**탐지 방법**
1. `/admin`, `/dashboard`, `/api/admin/*` 경로에 인증 없이 접근 시도
2. robots.txt에서 admin 경로 노출 여부 확인

**자동화 가능?**
예, URL 목록 기반 접근 테스트.

---

### VULN-012 — 로그인 확인용 암호가 너무 쉬워요

**탐지 방법**
1. 코드에서 JWT sign 호출 시 사용되는 secret 값 확인
2. 약한 키 목록: `secret`, `password`, `1234`, `jwt-secret`, `changeme`
3. 키 길이가 32자 미만인 경우 경고

**자동화 가능?**
예, 코드 패턴 검색.

---

### VULN-013 — 로그인 상태가 영원히 유지돼요

**탐지 방법**
1. JWT 생성 코드에서 `expiresIn` 또는 `exp` 클레임 확인
2. 세션 쿠키의 `maxAge` 설정 확인
3. 만료 없거나 30일 이상이면 경고

**자동화 가능?**
예, 코드 검색.

---

### VULN-014 — 로그인 시도 횟수 제한이 없어요

**탐지 방법**
1. 로그인 API에 rate limiter 미들웨어 존재 여부 확인
2. 실제로 10회 이상 실패 시도 후 차단 여부 테스트

**자동화 가능?**
예, 자동 요청 테스트.

---

### VULN-015 — 이메일 확인 없이 가입할 수 있어요

**탐지 방법**
1. 회원가입 플로우에서 이메일 인증 단계 유무 확인
2. Supabase/Firebase auth 설정에서 email verification 활성화 여부

**자동화 가능?**
부분적. 설정 확인은 수동.

---

### VULN-016 — 탈퇴 기능이 없어요

**탐지 방법**
1. UI에서 계정 삭제/탈퇴 버튼 존재 여부 확인
2. API에서 DELETE /account 또는 유사 엔드포인트 존재 여부

**자동화 가능?**
부분적. UI 크롤링 + API 검색.

---

### VULN-017 — 비밀번호를 브라우저에 그대로 저장하고 있어요

**탐지 방법**
1. 코드에서 `localStorage.setItem` 또는 `sessionStorage.setItem` 검색
2. 저장하는 값이 password, token, secret, key 관련인지 확인

**자동화 가능?**
예, 코드 검색.

---

### VULN-018 — 소셜 로그인 설정이 잘못돼 있어요

**탐지 방법**
1. OAuth 설정에서 redirect URI가 와일드카드 또는 localhost인지 확인
2. Google/GitHub OAuth 앱 설정에서 허용 도메인 확인

**자동화 가능?**
부분적. 설정 파일 확인은 가능, 외부 서비스 설정은 수동.

---

### VULN-061 — 관리자와 일반 사용자 구분이 없어요

**탐지 방법**
1. 사용자 모델에 role/permission 필드 존재 여부 확인
2. API 라우트에서 역할 기반 접근 제어 로직 확인

**자동화 가능?**
부분적. 코드 분석 필요.

---

## DATA 카테고리

### VULN-019 — 데이터 열람 권한이 꺼져 있어요

**탐지 방법**
1. Supabase 대시보드 > Authentication > Policies 확인
2. `supabase_migration` 파일에서 `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` 존재 여부
3. Supabase CLI: `supabase inspect db policies`

**자동화 가능?**
예, Supabase Management API로 확인 가능.

---

### VULN-020 — 데이터베이스 규칙이 '모두 허용'이에요

**탐지 방법**
1. `firebase.rules` 또는 `firestore.rules` 파일에서 `allow read, write: if true` 검색
2. `allow read, write` 뒤에 조건이 없거나 `if true`인 경우

**자동화 가능?**
예, 파일 패턴 검색.

---

### VULN-021 — 데이터베이스 비밀번호가 초기값이에요

**탐지 방법**
1. DB 접속 문자열에서 password 부분 추출
2. 약한 비밀번호 목록과 대조: `postgres`, `root`, `admin`, `password`, `1234`, `changeme`

**자동화 가능?**
예, 문자열 매칭.

---

### VULN-022 — 고객 정보가 잠금 없이 보관돼 있어요

**탐지 방법**
1. DB 스키마에서 email, phone, name 등 개인정보 필드 확인
2. 해당 필드가 평문(text/varchar)인지, 암호화 함수로 감싸져 있는지 확인

**자동화 가능?**
부분적. 스키마 분석 자동화 가능.

---

### VULN-023 — 백업(복사본)이 하나도 없어요

**탐지 방법**
1. DB 호스팅 서비스(Supabase, PlanetScale 등)에서 자동 백업 설정 확인
2. 수동 백업 스크립트 또는 cron job 존재 여부

**자동화 가능?**
부분적. 서비스 API로 확인 가능한 경우도 있음.

---

### VULN-024 — 누구나 데이터를 한꺼번에 내보낼 수 있어요

**탐지 방법**
1. 데이터 내보내기/다운로드 API에 인증 및 rate limit 확인
2. 한 번 요청으로 전체 데이터를 가져올 수 있는지 테스트

**자동화 가능?**
예, API 호출 테스트.

---

### VULN-025 — 탈퇴한 고객 데이터가 그대로 남아있어요

**탐지 방법**
1. 회원 탈퇴 API에서 관련 테이블 DELETE/soft-delete 로직 확인
2. 탈퇴 후 해당 사용자 데이터 조회 가능 여부 테스트

**자동화 가능?**
부분적.

---

### VULN-026 — 개인정보처리방침이 없어요

**탐지 방법**
1. 사이트 footer 또는 /privacy, /terms 경로에 페이지 존재 여부 확인
2. 개인정보 수집 폼이 있는데 방침 링크가 없는 경우 경고

**자동화 가능?**
예, URL 존재 확인.

---

### VULN-062 — 필요 이상으로 개인정보를 많이 모으고 있어요

**탐지 방법**
1. 회원가입/설정 폼에서 수집하는 필드 목록 확인
2. 서비스에 불필요한 주민번호, 주소, 생년월일 등 필드가 있는지

**자동화 가능?**
부분적. 폼 필드 분석.

## NETWORK 카테고리

### VULN-027 — 아무 사이트에서나 우리 서버에 요청을 보낼 수 있어요

**탐지 방법**
1. `curl -I https://target.com/api/... | grep Access-Control-Allow-Origin`
2. 값이 `*`이면 위험
3. 서버 코드에서 `cors({ origin: '*' })` 또는 유사 설정 검색

**자동화 가능?**
예, HTTP 헤더 검사.

---

### VULN-028 — 한 사람이 무한으로 요청을 보내도 막지 못해요

**탐지 방법**
1. 서버 코드에서 rate limiter 미들웨어 존재 여부 확인
2. 동일 IP/토큰으로 100회 연속 요청 시 429 응답 여부 테스트

**자동화 가능?**
예, 자동 요청 테스트.

---

### VULN-029 — 외부 스크립트가 마음대로 들어올 수 있어요

**탐지 방법**
1. `curl -I https://target.com | grep Content-Security-Policy`
2. CSP 헤더가 없거나 `unsafe-inline`, `unsafe-eval`이 있는 경우

**자동화 가능?**
예, HTTP 헤더 검사.

---

### VULN-030 — 대량 접속 공격에 무방비 상태예요

**탐지 방법**
1. Cloudflare, AWS WAF, Vercel 방화벽 등 DDoS 방어 서비스 사용 여부
2. DNS 조회로 CDN/프록시 사용 여부 확인

**자동화 가능?**
부분적. DNS 조회 자동화 가능.

---

### VULN-031 — 서버 내부 정보가 밖에서 보여요

**탐지 방법**
1. 응답 헤더에서 `X-Powered-By`, `Server`, `X-AspNet-Version` 등 확인
2. 이런 헤더가 노출되면 경고

**자동화 가능?**
예, 헤더 검사.

---

### VULN-032 — 외부 알림을 진짜인지 확인 안 하고 받아요

**탐지 방법**
1. 웹훅 엔드포인트 코드에서 서명 검증 로직 확인
2. Stripe: `stripe.webhooks.constructEvent()` 사용 여부
3. GitHub: `X-Hub-Signature-256` 검증 여부

**자동화 가능?**
부분적. 코드 분석 필요.

---

### VULN-033 — 방문자를 엉뚱한 곳으로 보낼 수 있어요

**탐지 방법**
1. 리다이렉트 파라미터(`?redirect=`, `?next=`, `?url=`)가 있는 URL 검색
2. 해당 파라미터에 외부 URL을 넣어 테스트

**자동화 가능?**
예, URL 파라미터 테스트.

## INJECTION 카테고리

### VULN-034 — 입력란에 명령어를 넣으면 데이터가 나와요

**탐지 방법**
1. 코드에서 SQL 쿼리가 템플릿 리터럴로 조립되는 부분 검색
2. 패턴: `` `SELECT * FROM ... WHERE ${userInput}` ``
3. Prepared statement / parameterized query 사용 여부 확인

**자동화 가능?**
