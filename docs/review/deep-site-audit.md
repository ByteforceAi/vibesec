# VibeSec / Byteforce 사이트 심층 비즈니스 분석

> 작성일: 2026-04-12
> 핵심 전제: "바이브코더가 AI를 못 믿어서 사람에게 맡기는 서비스"
> 비즈니스 모델: 1:1 수동 점검 + 보고서 납품 + 후속 관리

---

## 1. 현재 구조 진단 (페이지별 필요/불필요 판정)

### 1-1. 홈 (`+page.svelte`)

**현재 역할:** 스플래시 애니메이션(방패 + 바이너리 레인) -> 히어로 + 서비스 카드 3개(1:1 대면 점검, 긴급 점검, 월 관리) + 최근 스캔 내역. 하단 네비게이션으로 진단/리포트/요금제로 이동.

**판정: 유지하되 대폭 수정 필요**

문제점:
- "내 프로젝트 점검 받기" CTA가 `/diagnose`(자동 스캔 페이지)로 연결됨. 비즈니스 모델은 1:1 수동 점검인데, 첫 CTA가 자동 스캔으로 가는 것은 근본적 모순.
- "최근 점검" 영역이 가짜 스캔 이력을 보여줌. 실제 서비스에서 스캔 이력이 로컬 스토리지에만 저장되므로 의미 없음.
- 스플래시 3초가 첫 방문자의 이탈을 유발할 수 있음. 재방문 시 단축 로직은 있으나, 첫인상이 "테크 데모"처럼 보임.
- "3 agents ready" / "STANDBY" 등 개발자 감성의 상태바가 비개발자에게 불필요한 인지 부하를 줌.
- 하드코딩된 데모 스캔 결과(TeacherConnect)가 사이트에 노출됨. 실제 고객 프로젝트가 아님.

**필요한 변경:**
- 메인 CTA를 "상담 예약" (= `/incident`)으로 변경
- 스플래시를 1.5초 이하로 단축하거나, 재방문자에게는 완전 생략
- "최근 점검" 대신 "점검 사례" 또는 "보고서 미리보기"를 배치
- 상태바의 개발자 감성 요소 제거

---

### 1-2. 요금제 (`/packages/+page.svelte`)

**현재 역할:** 3단 요금제(기본 점검 39만 / 정밀 점검 89만 / 풀 정비 190만) + 추가 메뉴(긴급, 출장, 월 관리, 세부 단품) + 포함 항목 비교표 + 터미널 데모(.env 노출 시연) + 오시는 곳 + FAQ.

**판정: 핵심 페이지. 유지하되 구조 정리 필요**

강점:
- 가격이 명확하고 3단 비교가 직관적
- "뭐가 필요한지 모르겠으면 일단 오세요"라는 풋터 카피가 비즈니스 모델에 완벽히 부합
- FAQ 내용이 바이브코더의 실제 의문을 잘 반영

문제점:
- "무료 스캔으로 뭘 알 수 있나요?" FAQ가 자동 스캔 기능을 전제하고 있음. 비즈니스 모델과 충돌.
- "312개 세부 메뉴에서 필요한 것만"이라는 단품 안내가 있지만, 312개를 실제로 보여주는 것은 불필요. 상담 시 안내하는 것이 맞음.
- 터미널 데모(.env 노출)가 시각적으로 훌륭하지만, 비개발자에게 `OPENAI_API_KEY=sk-proj-4f8a...`는 의미 불명. 공포보다 혼란을 유발.
- CTA가 `/incident`(상담 예약)로 가는 것은 올바름.

---

### 1-3. 진단 (`/diagnose/+page.svelte`)

**현재 역할:** URL 입력 -> `fake-scanner.ts`가 랜덤 취약점을 뽑아서 가짜 스캔 결과 생성 -> `/report/[scanId]`로 이동.

**판정: 제거 또는 완전 재설계 필요**

이 페이지는 현재 비즈니스 모델의 핵심 모순:
- "사람이 직접 점검한다"가 가치인데, URL 넣으면 자동으로 결과가 나오면 "AI가 AI를 검사하는 꼴"이 됨
- `fake-scanner.ts`는 랜덤으로 취약점을 선택하므로 어떤 URL을 넣어도 비슷한 결과가 나옴. 신뢰를 쌓는 게 아니라 깎음.
- 실제 보안 스캔 없이 결과를 보여주면, 그 결과를 믿고 "안전하다"고 오판하는 위험이 있음
- 이 페이지의 존재 자체가 "사람 점검"의 가치를 희석함

**두 가지 선택지:**
- (A) 완전 제거: 이 자리에 "상담 접수 폼"을 배치
- (B) "간이 체크리스트"로 전환: 자동 스캔이 아닌, 사용자가 스스로 체크하는 5문항 자가진단. 결과가 "상담이 필요합니다"로 귀결.

권장: 선택지 B. 자가진단 → 상담 예약으로의 전환 장치로 활용.

### 1-4. 상담 (incident, `/incident/+page.svelte`)

**현재 역할:** 챗봇 형태의 상담 접수. 상담 유형 → 사용 도구 → 프로젝트 URL → 방문/온라인 선택 → 이메일 입력 → 접수 완료. 데이터는 localStorage에만 저장.

**판정: 가장 중요한 페이지. 유지하되 백엔드 연결 필수**

강점:
- 챗봇 UX가 부드럽고 단계가 자연스러움
- "어떤 도구로 만드셨나요?" 질문이 바이브코더 맥락에 정확히 맞음
- "서울/부산/온라인" 선택지가 실제 서비스 모델을 반영

치명적 문제:
- **데이터가 localStorage에만 저장됨.** 사용자가 브라우저를 닫으면 접수 데이터가 사라짐. 실제 비즈니스 운영 불가.
- "24시간 내로 연락드리겠습니다"라고 해놓고 실제 알림 메커니즘이 없음
- 이메일 수집 후 자동 확인 메일 발송 기능 없음

**필요한 변경:**
- 접수 데이터를 실제 백엔드(이메일 API, Google Form, Notion, Supabase 등)로 전송
- 접수 즉시 확인 이메일 자동 발송
- 관리자에게 Slack/카카오 알림 발송

---

### 1-5. 리포트 목록 (`/report/+page.svelte`)

**현재 역할:** fake-scanner의 스캔 이력 목록. localStorage 기반.

**판정: 현재 형태로는 불필요. 재설계 필요.**

이 페이지가 의미를 가지려면:
- 실제 점검이 완료된 고객에게 보고서를 열람하는 "고객 포탈" 역할이어야 함
- 현재는 fake scan 결과만 보여주므로 가치 없음

**미래 역할:** 로그인 기반 "내 점검 결과" 대시보드. 1차에서는 생략하고, 보고서를 PDF/이메일로 전달하는 것이 현실적.

---

### 1-6. 리포트 상세 (`/report/[scanId]/+page.svelte`)

**현재 역할:** fake scan 결과의 상세 뷰. 긴급/주의/안전 요약 + 발견 항목 리스트 + 추천 패키지(장바구니 추가).

**판정: "보고서 샘플"로 전환 가능**

이 페이지의 레이아웃 자체는 좋음. 문제는 fake scan 결과를 보여주는 것. 이것을 "실제 보고서는 이렇게 생겼습니다"라는 샘플 미리보기로 전환하면 강력한 전환 장치가 됨.

- 추천 패키지 → 장바구니 흐름은 제거 (온라인 결제 모델이 아니므로)
- 대신 "이런 보고서를 받게 됩니다 → 상담 예약" 흐름으로 전환

---

### 1-7. 결제 (`/checkout/+page.svelte`)

**현재 역할:** 장바구니 → mock 결제 (카드결제/카카오 상담). 실제 결제 기능 없음.

**판정: 제거**

이유:
- 39만~190만원 규모의 보안 점검을 온라인 카드결제로 즉시 구매하는 사람은 없음
- 이 가격대에서는 상담 → 견적 확인 → 계좌이체/카드결제가 자연스러움
- 장바구니에 "패키지 단품"을 담는 UX는 312개 카탈로그 e-commerce 모델의 잔재
- 결제 페이지의 존재 자체가 "사이트에서 바로 구매"라는 잘못된 기대를 만듦

---

### 1-8. 온보딩 (`/onboarding/+page.svelte`)

**현재 역할:** 4단계 온보딩 카드(환영 → 프로젝트 연결 → 무료 진단 → 결과 확인).

**판정: 제거 또는 극도로 단순화**

이유:
- 4단계 중 3, 4단계가 "무료 진단 → 결과 확인"이라는 자동 스캔 흐름을 전제함
- 비즈니스 모델이 1:1 수동 점검이면 이 온보딩은 잘못된 기대를 심어줌
- 랜딩에서 바로 가치를 전달하는 것이 앱 온보딩 4단계보다 효과적

---

### 1-9. 패키지 상세 (`/packages/[id]/+page.svelte`)

**현재 역할:** 312개 단품 패키지의 개별 상세 페이지. 증상, 설명, 포함 내역, 장바구니 추가.

**판정: 사이트에서 직접 노출 불필요**

이유:
- 비개발자가 "api-endpoint-hardening"이라는 패키지 이름을 보고 자기에게 필요한지 판단할 수 없음
- 312개 중 자기에게 맞는 것을 고르는 행위 자체가 비개발자에게 불가능
- "상담 시 안내"가 맞는 접근. 전문가가 점검 후 필요한 패키지를 추천하는 것이 비즈니스 모델에 부합

## 2. 제안하는 새 구조

### 현재 8개 페이지 → 5개로 재편

| 순서 | 경로 | 이름 | 역할 |
|------|------|------|------|
| 1 | `/` | 홈 | 첫인상 + 가치 전달 + 상담 CTA |
| 2 | `/packages` | 요금제 | 3단 요금제 + FAQ + 오시는 곳 |
| 3 | `/check` | 자가 진단 | 5문항 체크리스트 → "상담이 필요합니다" → 상담 예약 유도 |
| 4 | `/contact` | 상담 예약 | 현재 incident 페이지 유지 + 백엔드 연결 |
| 5 | `/sample` | 보고서 샘플 | 실제 보고서가 이렇게 생겼다는 미리보기 |

**제거 대상:**
- `/diagnose` (fake scanner) → `/check` (자가진단)으로 대체
- `/report`, `/report/[scanId]` → `/sample`로 통합
- `/checkout` → 완전 제거
- `/onboarding` → 완전 제거
- `/packages/[id]` → 사이트에서 비노출 (내부 관리용으로만 유지)

**네비게이션 재편:**

| 현재 | 변경 후 |
|------|---------|
| 홈 / 진단 / 리포트 / 요금제 | 홈 / 요금제 / 자가진단 / 상담 예약 |

## 3. 핵심 전환 장치 3가지

### 장치 1: 보고서 샘플 미리보기

**왜:** "돈을 내면 뭘 받는지"를 모르면 구매 결정이 불가능. 보고서가 핵심 상품이므로, 상품의 실물을 보여주는 것이 가장 강력한 전환 장치.

**구현 방법:**
- 실제 점검 사례 1건을 익명화하여 샘플 보고서로 제작
- 보고서 구성: (1) 3줄 요약 (2) 발견 항목 리스트 (심각도 + 비유 설명) (3) 수정 가이드 요약 (4) 수정 전/후 비교
- 홈에서 "실제 보고서 미리보기" 버튼으로 접근
- 보고서 하단에 "내 프로젝트도 점검 받기" CTA

**기대 효과:**
- "이 수준의 보고서를 39만원에 받을 수 있다"는 가치 인식
- 보고서의 비유 톤("문 앞에 열쇠를 놔둔 상태예요")을 직접 체험
- 경쟁사 대비 차별점(기술 용어 없는 보고서)을 증명

---

### 장치 2: 5문항 자가 진단 체크리스트

**왜:** 바이브코더는 "내 서비스가 위험한지 아닌지"를 모름. 자동 스캔 결과를 보여주면 "AI가 AI를 검사한 것"이라 신뢰가 없음. 대신 스스로 "예/아니오"로 체크하게 하면 자각이 생김.

**구현 방법:**
```
질문 1: 프로젝트에 .env 파일이 있나요? (모르겠어요 / 네 / 아니요)
질문 2: GitHub 레포가 public인가요? (모르겠어요 / 네 / 아니요)
질문 3: API 키를 코드에 직접 적은 적 있나요? (모르겠어요 / 네 / 아니요)
질문 4: 로그인 기능이 있나요? (없어요 / 있어요)
질문 5: Supabase/Firebase의 보안 규칙을 설정한 적 있나요? (모르겠어요 / 네 / 아니요)
```

결과 표시:
- "모르겠어요"가 2개 이상: "점검을 받아보시는 게 좋겠어요."
- "네"가 있으면서 "모르겠어요"도 있음: "몇 가지 확인이 필요합니다."
- 전부 안전: "기본은 잘 되어 있어요. 더 깊이 확인하고 싶으시면 상담 예약하세요."

**핵심:** 결과가 항상 "상담 예약"으로 귀결되되, 공포가 아닌 "확인해봅시다" 톤.

---

### 장치 3: 비용 대비 가치 시나리오

**왜:** 39만원이 비싼지 싼지 판단할 기준이 없음. "사고가 나면 얼마나 드는지"를 구체적 숫자로 보여주면 가격 저항이 줄어듦.

**구현 방법:**
요금제 페이지에 "만약 점검 안 하면?" 섹션 추가:

```
API 키 노출 → OpenAI 요금 폭탄: 평균 $500~$2,000 (70~280만원)
Supabase 데이터 유출 → 개인정보 사고: 과징금 최소 1,000만원
서비스 다운타임 1일 → 신뢰 손실: 측정 불가
```

vs.

```
기본 점검: 39만원, 2~3시간
정밀 점검: 89만원, 1~2일
```

마무리 카피: "점검 비용은 사고 비용의 10분의 1도 안 됩니다."

**주의:** 공포 마케팅이 아닌 "정보 제공" 톤으로. "해킹당합니다!" 대신 "이런 일이 실제로 일어나고 있어요."

## 4. 보고서 상품 설계안

### 4-1. 보고서 구성 (기본 점검 기준)

| 섹션 | 내용 | 분량 |
|------|------|------|
| 1. 한 줄 요약 | "총 5건의 확인 사항이 발견되었습니다. 이 중 1건은 즉시 조치가 필요합니다." | 1줄 |
| 2. 건강검진표 | 항목별 안전/주의/긴급 표시. 건강검진 결과표와 동일한 형태. | 1페이지 |
| 3. 발견 항목 상세 | 각 항목마다: 비유 설명 + 어디서 발견 + 왜 위험한지 + 어떻게 고치는지 | 항목당 1/2 페이지 |
| 4. 지금 당장 할 일 | 3가지 이내의 즉시 조치 목록. "이것만 먼저 하세요." | 1/2 페이지 |
| 5. 다음 단계 | 추가 점검/수정 가이드/월 관리 안내 | 1/2 페이지 |

### 4-2. 보고서 톤 예시

```
[긴급] 비밀번호가 문 앞에 놓여 있어요
```

## 5. 워크플로우 (접수 -> 점검 -> 보고서 -> 후속)

### 단계 1: 접수

| 항목 | 내용 |
|------|------|
| 사이트 역할 | 상담 예약 폼 (현재 `/incident` 페이지) |
| 수집 정보 | 상담 유형, 사용 도구, 프로젝트 URL, 방문/온라인 선택, 이메일, 연락처 |
| 자동화 | 접수 확인 이메일 즉시 발송 + 관리자 Slack/카카오 알림 |
| 수동 | 관리자가 24시간 내 연락하여 일정 확정 |

### 단계 2: 사전 확인

| 항목 | 내용 |
|------|------|
| 사이트 역할 | 없음 (이메일 또는 카카오톡으로 진행) |
| 수동 | GitHub 접근 권한 확인, 프로젝트 규모 파악, 견적 확정 |
| 자동화 | 일정 확정 이메일 (일시 + 장소/Zoom 링크 + 사전 준비 안내) |

### 단계 3: 점검 실시

| 항목 | 내용 |
|------|------|
| 사이트 역할 | 없음 (대면 또는 화상으로 진행) |
| 수동 | 엔지니어가 직접 코드 리뷰 + 보안 체크리스트 실행 |
| 소요 시간 | 기본 2~3시간 / 정밀 1~2일 / 풀 3~7일 |

### 단계 4: 보고서 전달

| 항목 | 내용 |
|------|------|
| 사이트 역할 | (1차) 없음 - PDF 이메일 전달. (중기) 웹 대시보드에서 열람. |
| 자동화 | 보고서 PDF 생성 템플릿 + 이메일 자동 발송 |
| 수동 | 보고서 내용 작성, 수정 가이드 포함 |

### 단계 5: 후속 관리

| 항목 | 내용 |
|------|------|
| 사이트 역할 | (중기) 고객 포탈에서 재점검 요청 |
| 자동화 | 보증 기간 만료 알림 이메일 (14일/30일) |
| 수동 | 재점검 실시, 월 관리 계약 갱신 |

### 워크플로우 요약

```
[사이트] 상담 예약  ←-- 유일하게 사이트가 담당하는 핵심 전환점
   ↓
[이메일] 접수 확인 자동 발송
   ↓
[수동] 관리자 연락 → 일정 확정 → 견적 확인
   ↓
[이메일] 일정 확정 + 준비 안내
   ↓
[수동] 점검 실시 (대면/화상)
   ↓
[수동+자동] 보고서 작성 → PDF 생성 → 이메일 발송
   ↓
[자동] 보증 기간 알림 → 재점검 제안
```

## 6. 즉시 실행 가능한 5가지 (우선순위 순)

### 1순위: 상담 접수 백엔드 연결 (소요 1~2일)

현재 `/incident` 페이지의 접수 데이터가 localStorage에만 저장됨. 이것은 비즈니스 운영의 가장 기본적인 단절.

실행 방법:
- 최소: Google Forms로 데이터 전송 (무료, 즉시 가능)
- 권장: Supabase 테이블에 저장 + 관리자 이메일 알림
- 이상: Supabase + Resend(이메일 API)로 고객 확인 메일 자동 발송

### 2순위: 홈 CTA를 "상담 예약"으로 변경 (소요 30분)

`goto(`${base}/diagnose`)` → `goto(`${base}/incident`)`

현재 "내 프로젝트 점검 받기" 버튼이 자동 스캔 페이지로 가는 것을 상담 예약 페이지로 변경. 이 한 줄 변경이 전체 사이트의 전환 방향을 바로잡음.

### 3순위: 보고서 샘플 1건 제작 + 노출 (소요 3~5일)

실제 점검 사례 1건(익명화)을 위 4장의 설계안 기준으로 작성. `/sample` 경로에 배치. 홈과 요금제 페이지에서 링크.

### 4순위: `/diagnose`를 자가 진단 체크리스트로 전환 (소요 2~3일)

fake-scanner 제거. 5문항 체크리스트 UI로 교체. 결과가 "상담 예약" CTA로 연결.

### 5순위: FAQ 내용 정비 (소요 1시간)

"무료 스캔으로 뭘 알 수 있나요?"를 "점검 받으면 뭘 알 수 있나요?"로 변경. 자동 스캔을 전제한 모든 문구를 1:1 수동 점검 기준으로 수정.

## 결론

현재 사이트의 가장 큰 문제는 **자동 스캔 SaaS 모델의 잔재**와 **1:1 수동 점검 모델**이 혼재되어 있다는 것. 방문자는 "여기서 뭘 하라는 건지"가 불명확함.

해결의 핵심은 단순함: **사이트의 모든 길이 "상담 예약"으로 통하게** 만들면 됨.

- 홈 → 상담 예약
- 자가 진단 → "확인이 필요합니다" → 상담 예약
- 보고서 샘플 → "이런 보고서 받고 싶다" → 상담 예약
- 요금제 → "이 가격이면 괜찮다" → 상담 예약

worldbuilding.md의 "동네 대장장이" 세계관은 훌륭함. 이 세계관이 사이트 구조와 UX에 그대로 반영되면, "뭐가 필요한지 모르겠으면 일단 오세요. 보고 말씀드리겠습니다."라는 풋터 카피가 사이트 전체의 경험이 됨.
