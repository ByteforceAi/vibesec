## 2. 퍼널 구조 분석

### 현재 퍼널

```
방문 (홈)
  |  예상 100%
  v
무료 진단 시작 (URL 입력)
  |  예상 전환율: 40~60%  ← "무료"가 강력한 미끼
  v
진단 중 (로딩)
  |  이탈: 10~15% (기다림 못 참는 사람)
  v
건강검진표 (리포트)
  |  예상 전환율: 70~80% (여기까진 무료니까 본다)
  v
추천 처방 → 상담 예약
  |  예상 전환율: 5~15%  ← 병목
  v
결제 (39만~190만원)
  |  예상 전환율: 30~50% (상담까지 온 사람 중)
```

### 핵심 병목: 리포트 → 상담 전환 (5~15%)

원인 분석:
- 무료 진단 결과를 보고 "아 이 정도면 괜찮네" 하고 떠나는 사람이 대다수
- 상담 예약 페이지(/incident)가 "긴급 상담" 프레이밍이라 일반 사용자가 들어가기 부담스러움
- 리포트에서 "이걸 안 고치면 구체적으로 뭐가 일어나는지" 시나리오가 없음
- 가격을 보기 전에 상담을 예약해야 하는 구조 (가격 투명성 부족)

### 누락된 퍼널 단계

1. **리드 수집 지점**: 무료 진단 결과를 보여주되, "상세 리포트를 이메일로 받기"를 넣으면 이메일 수집 가능
2. **너지 (Nudge)**: 진단 후 7일 뒤 "그때 발견된 문제가 아직 남아있어요" 이메일
3. **소셜 프루프**: "이번 달 47개 프로젝트를 점검했어요" 같은 신뢰 지표

## 3. 대기자 목록 최적화

### 현재 상태

대기자 목록 전략을 채택했으나 구현이 미비하다. /diagnose 페이지에서 URL만 받고 있고, 이메일 수집 지점이 없다.

### 권고: 2단계 수집 설계

**1단계 (무료 진단 직후, /report 화면)**

수집 항목:
- 이메일 (필수) - "검진 결과를 이메일로 보내드릴게요"
- 프로젝트 URL (이미 보유)

이 시점에 수집하는 이유: 사용자가 결과를 받았을 때 가치를 느끼는 순간이므로 이메일 제출 의향이 가장 높다.

**2단계 (상담 예약 시, /incident 화면)**

추가 수집 항목:
- 사용 도구 (Cursor / v0 / Lovable / bolt.new / 기타) - 드롭다운
- 긴급도 (지금 급해요 / 이번 주 안에 / 여유 있어요) - 3단 선택
- 한 줄 상황 설명 (선택) - "결제 기능 도입 전이에요" 같은 컨텍스트

이 시점에 수집하는 이유: 상담 예약까지 온 사람은 의지가 있으므로 추가 정보를 줄 용의가 있다. 이 데이터로 상담 전 사전 분석이 가능해져 상담 품질이 올라간다.

### 대기자 목록 구체 설계안

```
[진단 완료]
    |
    v
"검진 결과를 이메일로 보내드릴까요?"
    |
  [이메일 입력] ← 리드 수집 1차
    |
    v
"상세 리포트 + 예상 견적을 보내드려요"
    |
    v
[이메일 도착: PDF 리포트 + "지금 예약하면 이번 주 상담 가능"]
    |
    v
7일 후 팔로업: "아직 고치지 않으셨다면, 무료 재진단 해드릴게요"
    |
    v
30일 후 최종: "한 달 전 발견된 문제 상태 업데이트"
```

이메일 시퀀스 목표:
- 1차 (즉시): 리포트 전달 + 가치 제공
- 2차 (7일): 재진단 유도 + 긴급도 상기
- 3차 (30일): 상태 변화 알림 + 월 관리 구독 소개

## 4. 가격 심리 분석

### 현재 가격 구조

| 패키지 | 가격 | 역할 |
|--------|------|------|
| 기본 점검 | 39만원 | 진입 |
| 정밀 점검 | 89만원 | 메인 (추천 표시) |
| 풀 정비 | 190만원 | 프리미엄 |
| 긴급 점검 | 49만원 | 긴급 |
| 월 관리 | 59만원/월 | 리텐션 |

### 문제점

1. **39만원은 첫 접점으로 높다**: 페르소나 P1(사이드프로젝트 기획자)의 예산 감수성이 "무료~5만원"으로 설정되어 있다. 39만원을 보는 순간 이탈할 가능성이 높다.

2. **무료 진단과 39만원 사이의 간극**: 무료 → 39만원은 점프가 크다. 중간 단계가 없다.

3. **89만원에 "추천" 표시**: 앵커링 전략으로는 맞지만, P1이 89만원을 보면 "나한테는 아닌 것 같다"고 판단한다.

### 권고: "미끼 상품" 도입

```
무료 진단 (현재 있음)
    |
    v
[신규] 간단 수리: 9만9천원 (1가지 문제 수정 + 확인)
    |
    v
기본 점검: 39만원 (현재 유지)
    |
    v
정밀 점검: 89만원 (현재 유지)
    |
    v
풀 정비: 190만원 (현재 유지)
```

9만9천원의 역할:
- P1이 "이것 하나만 고치면 돼요" 추천을 받았을 때 바로 결제할 수 있는 가격대
- "첫 결제" 허들을 낮춤. 한 번 결제한 고객은 재구매 확률이 5배 이상
- 312개 세부 패키지 중 단일 항목(39,000원~)이 이미 존재하지만, /packages 페이지에서 메인 플랜과 분리되어 있어 발견이 어렵다

대안으로 "5분 무료 전화 상담"도 고려할 수 있으나, 1:1 모델에서 무료 상담은 시간 비용이 크다. 차라리 진단 리포트의 품질을 높여서 "이 리포트만으로도 가치가 있다"고 느끼게 하는 편이 낫다.

## 5. 경쟁 포지셔닝

### "바이브코딩 전문"의 강점

- 2026년 4월 현재, "바이브코딩 보안"을 전문으로 내세우는 경쟁자가 거의 없다
- Cursor, v0, bolt.new 사용자가 빠르게 늘고 있어 타이밍이 좋다
- "컴퓨터 수리점" 메타포가 비개발자에게 직관적이다

### 취약점

1. **모방 용이**: 기술적 해자(moat)가 없다. 보안 컨설팅은 진입장벽이 낮다.
2. **AI 도구 자체의 보안 개선**: Cursor, v0 등이 보안 기능을 내장하면 수요가 줄 수 있다.
3. **가격 경쟁**: 프리랜서 개발자가 "바이브코딩 보안 점검 10만원"으로 진입 가능.

### 방어 전략 권고

1. **312개 패키지 데이터를 콘텐츠 자산화**: 각 패키지의 "증상 → 원인 → 해법" 콘텐츠를 블로그/유튜브로 공개. 검색 유입 확보 + 전문성 증명
2. **자동화 도구 선점**: 무료 진단 도구의 정확도와 범위를 경쟁자가 따라오기 어렵게 키우기
3. **커뮤니티 구축**: "바이브코딩으로 만든 사람들" 카카오톡 오픈채팅 또는 디스코드. 여기서 자연스럽게 보안 상담 연결
4. **실적 공개**: "이번 달 점검한 프로젝트 수", "발견한 문제 수" 등을 투명하게 공개하여 신뢰 구축

## 6. 리텐션 분석

### 현재 경로

```
첫 점검 (39~190만원, 1회성)
    |
    ? (연결 없음)
    v
월 관리 (59만원/월)
```

### 문제점

- 첫 점검에서 월 관리로의 전환 유도가 설계되어 있지 않다
- /packages 페이지에서 월 관리는 "추가 메뉴" 섹션에 묻혀 있다
- 풀 정비(190만원)에만 "30일 모니터링 포함"이 있는데, 30일 후 자동으로 월 관리를 제안하는 경로가 없다

### 권고: "맛보기 구독" 전략

```
풀 정비 완료 (30일 모니터링 포함)
    |
    v
30일 후: "모니터링 기간이 끝나요. 계속하시겠어요?"
    |
    v
[월 관리 첫 달 50% 할인: 29만5천원]
    |
    v
2개월 차부터 정상가 59만원
```

추가 리텐션 장치:
- 모든 유료 점검 고객에게 "분기별 재진단" 제안 이메일
- 월 관리 고객 전용 "긴급 대응 SLA 4시간" 혜택 강조
- "추천인 할인": 기존 고객이 다른 사람을 소개하면 월 관리 1개월 무료

## 7. 확장 가능성 분석

### 1:1 대면 모델의 한계

- 마곡 사무실 기준 하루 최대 2~3건 (2~3시간/건)
- 화상 포함해도 월 40~60건이 물리적 한계
- 평균 객단가 80만원 가정 시 월 매출 상한: 3,200만~4,800만원
- 1인 운영 시 실질 가용: 월 15~20건 = 1,200만~1,600만원

### 자동화 가능 영역

| 현재 수동 | 자동화 가능 여부 | 우선순위 |
|----------|---------------|---------|
| 무료 진단 (URL 스캔) | 이미 자동화됨 (fake-scanner 모듈 존재) | 완료 |
| 진단 리포트 생성 | 자동화 가능. 312개 패키지 데이터와 매칭하면 자동 추천 | 높음 |
| 상담 예약/일정 관리 | 캘린들리 등 연동으로 자동화 | 높음 |
| 견적서 생성 | 진단 결과 기반 자동 견적 가능 | 중간 |
| 간단한 수정 작업 | 일부 (env 파일 정리, RLS 설정 등) 템플릿화 가능 | 낮음 (품질 리스크) |
| 월 관리 모니터링 | 스캔 자동화 + 알림 시스템 | 높음 |
| 리포트 PDF 생성 | 완전 자동화 가능 | 높음 |

### 스케일 로드맵

1단계 (0~3개월): 자동 진단 + 리포트 자동 생성으로 "무료~9만9천원" 구간을 셀프서비스화
2단계 (3~6개월): 월 관리 모니터링 자동화로 반복 매출(MRR) 확보
3단계 (6~12개월): 다른 보안 엔지니어를 네트워크로 연결하여 "바이브코딩 보안 전문가 플랫폼" 확장

## 8. 312개 패키지 활용 전략

### 현재 활용 상태

- /packages 페이지에서 "세부 단품: 39,000원~" 한 줄로만 언급
- content/packages/index.json에 312개가 구조화되어 있지만 UI에서 탐색 불가
- 8개 카테고리(SECRETS 41, RESPONSE 41, AUTH 40, DATA 40, NETWORK 40, INJECTION 40, INFRA 35, MONITOR 35)로 균형 있게 분포
- 무료 8개, 유료 304개, 평균 가격 179,151원

### 활용 방안 5가지

**1. 진단 결과와 패키지 자동 매칭**
- 무료 진단에서 발견된 문제와 312개 패키지를 자동 매칭
- "이 문제를 고치려면 이 패키지가 필요해요" 형태로 추천
- 이렇게 하면 "39만원짜리 기본 점검"이 아니라 "4만9천원짜리 이것 하나만"이라는 선택지가 생김

**2. 콘텐츠 마케팅 자산**
- 312개 패키지 각각을 블로그 글 하나로 전환: "바이브코딩으로 만든 앱에서 [증상명]이 발견되면?"
- SEO 키워드: "cursor 보안", "v0 api키 노출", "supabase rls 설정" 등
- 312개 글 = 312개 검색 유입 경로

**3. 셀프서비스 가격 계산기**
- 사용자가 증상을 체크하면 필요한 패키지 목록 + 총 견적이 자동 산출
- "세부 단품으로 하면 12만7천원, 기본 점검으로 하면 39만원" 비교 표시
- 이 비교가 기본 점검의 가치를 정당화함

**4. 긴급도 기반 우선순위 표시**
- 312개 중 severity가 high인 항목을 진단 결과 상단에 배치
- "지금 바로 고쳐야 할 것 2개" + "이번 주 안에 하면 좋은 것 5개" + "나중에 해도 되는 것 12개"
- 이 구분이 가격 납득을 돕는다

**5. 구독형 번들 설계 근거**
- 월 관리 구독에 매달 어떤 패키지를 점검하는지 명시
- "월 59만원에 매달 35개 항목 정기 점검" → 개별 구매 시 140만원 상당
- 이 비교가 구독의 가치를 명확하게 만든다

## 9. 개선 권고 5가지 (우선순위 순)

### 권고 1: 이메일 수집 지점 추가 [긴급]

**현재**: 무료 진단 결과를 보여주고 끝. 리드가 빠져나간다.
**개선**: 리포트 화면에서 "상세 결과를 이메일로 받기" 게이트 추가.
**기대 효과**: 무료 진단 사용자의 30~50%에서 이메일 수집. 이후 이메일 시퀀스로 상담 전환 가능.
**구현 난이도**: 낮음 (UI 변경 + 이메일 서비스 연동)

### 권고 2: 저가 진입 상품(9만9천원) 신설 [높음]

**현재**: 무료 → 39만원의 간극이 크다.
**개선**: 진단 결과에서 가장 급한 문제 1개를 9만9천원에 수정하는 "긴급 수리" 상품 추가. 312개 패키지 중 단일 항목을 묶어서 제공.
**기대 효과**: P1(예산 감수성 높은 기획자)의 첫 결제 허들을 낮춤. 첫 결제 고객의 재구매율이 신규보다 5배 이상 높다.
**구현 난이도**: 중간 (가격 체계 재설계 + UI 변경)

### 권고 3: 진단 결과 → 패키지 자동 매칭 [높음]

**현재**: 진단 결과와 312개 패키지가 연결되지 않는다.
**개선**: 진단에서 발견된 문제 유형을 312개 패키지의 category/severity와 매칭하여 "이 문제엔 이 수리가 필요해요" 자동 추천.
**기대 효과**: 리포트 → 상담 전환율 2~3배 개선. "뭘 해야 하는지 모르겠다"를 "이것 하나만 하면 된다"로 전환.
**구현 난이도**: 중간 (매칭 로직 + UI)

### 권고 4: 월 관리 전환 경로 설계 [중간]

**현재**: 풀 정비의 30일 모니터링이 끝나면 연결이 끊긴다.
**개선**: 모니터링 종료 7일 전 "계속하시겠어요?" 알림 + 첫 달 50% 할인 제안. 모든 유료 점검 결과 하단에 "정기 관리하면 이런 문제를 미리 잡아요" 섹션 추가.
**기대 효과**: 유료 고객의 20~30%가 월 관리로 전환하면 MRR 안정화. 월 10명이면 월 590만원 반복 매출.
**구현 난이도**: 중간 (이메일 시퀀스 + UI)

### 권고 5: 콘텐츠 마케팅 시작 [중간]

**현재**: 유입 경로가 검색과 지인 추천뿐이다.
**개선**: 312개 패키지를 기반으로 주 2회 블로그 글 발행. 제목 예시: "Cursor로 만든 앱에서 API 키가 노출되면 벌어지는 일", "Supabase를 쓴다면 이것만은 꼭 확인하세요"
**기대 효과**: 3개월 후 월 5,000~10,000 검색 유입. 무료 진단 전환율 적용 시 월 50~100건 리드.
**구현 난이도**: 낮음 (콘텐츠 작성만, 코드 변경 없음)
