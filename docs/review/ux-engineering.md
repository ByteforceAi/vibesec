## 2. 핵심 플로우 분석

### 2.1 주요 사용자 여정: 방문 → 관심 → 예약

```
[첫 방문]
    │
    ├─ localStorage 체크 → vibesec_visited 없음
    │      │
    │      └─ /onboarding (4단계 스와이프)
    │              │
    │              └─ 완료 → /diagnose
    │
    ├─ vibesec_visited 있음 → / (홈)
    │      │
    │      ├─ "레포 점검 시작" CTA → /diagnose
    │      ├─ "상담 예약" (헤더) → /incident
    │      └─ 최근 진단 클릭 → /report/[scanId]
    │
    └─ 하단 네비 → /diagnose | /report | /packages
```

### 2.2 진단 → 결제 플로우

```
/diagnose
    │  URL 입력 + "점검 시작"
    │  프로그레스 애니메이션 (mock)
    │
    └─ 완료 → /report/[scanId]
           │
           ├─ 발견 항목 클릭 → 바텀시트 상세
           │      └─ 추천 패키지 버튼 → /packages/[id]
           │
           ├─ 추천 패키지 "+" 버튼 → 장바구니 추가
           │      └─ "결제하기" → /checkout
           │
           └─ "전체 패키지 보기" → /packages
                  │
                  └─ (요금제 페이지는 "상담 예약" 모델)
                         └─ CTA → /incident
```

### 2.3 클릭 수 분석

| 여정 | 클릭 수 | 판정 |
|------|:-------:|------|
| 첫 방문 → 진단 시작 | 5 (온보딩 4단계 + URL 입력) | 과다. 온보딩이 강제 |
| 홈 → 진단 → 리포트 확인 | 3 | 적정 |
| 리포트 → 패키지 구매 | 3~4 | 적정 |
| 홈 → 상담 예약 | 2 (헤더 버튼 → 증상 선택 → 제출) | 적정 |
| 요금제 → 상담 예약 | 2 | 적정 |
| 요금제 → 특정 단품 구매 | 불가능 | 문제. 요금제 페이지에서 312개 단품으로의 경로 없음 |

### 2.4 이탈 가능성이 높은 지점

1. **온보딩 강제 리다이렉트**: 첫 방문 시 홈을 보기도 전에 `/onboarding`으로 보냄. 사용자가 검색/링크로 직접 도착했을 때 맥락 상실
2. **요금제 ↔ 단품 패키지 단절**: `/packages` 페이지는 3개 컨설팅 플랜(39만/89만/190만) 중심인데, `/packages/[id]`는 312개 단품 상세. 이 두 세계를 연결하는 UI 없음
3. **결제 페이지 진입 후 빈 장바구니**: 직접 URL로 `/checkout` 접근 시 빈 장바구니 상태. "패키지 보기" CTA가 `/packages`로 보내지만, 거기서 장바구니에 담는 기능이 없음
4. **리포트 상세 → 결제 경로의 mock 의존**: `recommendedPackageIds`가 빈 배열이면 추천 패키지 섹션 자체가 렌더링 안 됨. 진단→구매 전환 경로가 사라짐

## 3. 네비게이션 구조 분석

### 3.1 현재 구조

```
[상단 바] ─── 모든 페이지에 존재 (페이지별 복붙)
│   ├─ 브랜드명: "Byteforce Security"
│   └─ "상담 예약" 버튼 → /incident
│       (리포트 상세/결제/긴급/패키지 상세에서는 "뒤로가기" 버튼으로 대체)

[하단 네비] ─── 홈, 진단, 리포트에만 존재
│   ├─ 홈 → /
│   ├─ 진단 → /diagnose
│   ├─ 리포트 → /report
│   └─ 요금제 → /packages

[하단 네비 없음] ─── 요금제, 결제, 긴급, 온보딩, 패키지 상세, 리포트 상세
```

### 3.2 문제점

- **네비게이션 불일치**: 하단 네비가 있는 페이지(3개)와 없는 페이지(6개)가 혼재. 사용자가 요금제 페이지에서 진단 페이지로 돌아가려면 브라우저 뒤로가기만 가능
- **요금제 페이지의 이중 네비**: 사이드바(데스크톱) + 세그먼트 컨트롤(모바일)이 있지만, 하단 네비는 없음. 다른 페이지로의 이동 수단이 없음
- **"상담 예약"의 위치**: 모든 페이지 상단 바 오른쪽에 고스트 버튼으로 존재. 시각적 가중치가 낮아 핵심 CTA치고 눈에 띄지 않음
- **뒤로가기 불일치**: `/report/[scanId]`는 `goto('/report')`로, `/checkout`은 `history.back()`으로, `/packages/[id]`는 `goto('/packages')`로 뒤로감. 동작이 예측 불가

### 3.3 권고

하단 네비를 **공유 컴포넌트**로 추출하고, "상담 예약"을 5번째 탭으로 승격하거나 FAB(플로팅 액션 버튼)으로 전환해야 함. 현재 모델에서 상담 예약이 핵심 전환 동작이므로, 모든 페이지에서 1탭으로 접근 가능해야 함.

## 4. 정보 아키텍처 분석

### 4.1 현재 페이지 구조 (9개)

```
/                     ← 홈 (대시보드)
/onboarding           ← 첫 방문 온보딩 (4단계)
/diagnose             ← URL 입력 + 스캔
/report               ← 리포트 목록
/report/[scanId]      ← 리포트 상세
/packages             ← 요금제 (3개 플랜 + 추가 메뉴 + FAQ + 위치)
/packages/[id]        ← 단품 패키지 상세 (312개 중 1개)
/checkout             ← 장바구니 + 결제
/incident             ← 긴급 상담 (증상 선택 + 연락)
```

### 4.2 불필요하거나 합칠 수 있는 페이지

| 페이지 | 판정 | 이유 |
|--------|------|------|
| `/onboarding` | **합치거나 제거** | 4단계가 텍스트만 보여줌. 홈 페이지의 빈 상태로 충분히 대체 가능. 강제 리다이렉트는 UX 마찰 |
| `/packages/[id]` | **역할 재정의 필요** | 312개 단품 상세인데, `/packages` 페이지(3개 플랜)와 연결 고리가 없음. 이 페이지로의 진입점이 리포트 상세의 추천에만 의존 |
| `/checkout` | **유지하되 진입 경로 정리** | 현재 장바구니 모델과 컨설팅 모델이 공존. 어느 쪽인지 결정 필요 |
| `/incident` | **요금제와 통합 검토** | "상담 예약"이 실질적으로 모든 플랜의 CTA. 별도 페이지가 아니라 요금제 페이지 내 섹션 또는 모달로 충분 |

### 4.3 비즈니스 모델 충돌

현재 사이트는 두 가지 판매 모델이 혼재되어 있음:

**모델 A: 컨설팅 예약 (요금제 페이지)**
- 3개 플랜 (기본 점검 39만 / 정밀 점검 89만 / 풀 정비 190만)
- 모든 CTA가 `/incident` (상담 예약)으로 향함
- 수리점 비유에 맞는 자연스러운 모델

**모델 B: 단품 장바구니 (리포트 상세 → 패키지 상세 → 결제)**
- 312개 세부 패키지를 개별 장바구니에 담아 결제
- `addToCart`, `cartItems`, `calculateTotal` 등 이커머스 인프라 구축됨
- 비개발자 타겟에게 312개 메뉴는 압도적

**권고**: 모델 A (컨설팅 예약)를 주 모델로 확정. 모델 B는 "세부 단품"으로 상담 과정에서 제안하는 보조 모델로 격하. UI에서는 모델 A만 전면에 노출.

## 5. 모바일 UX 분석

### 5.1 요금제 페이지 사이드바 소실

`@media (max-width: 768px)`에서 `.side { display: none; }` 처리. 모바일에서:
- 사이드바의 추가 메뉴(긴급 점검, 출장 점검, 월 관리, 세부 단품) 가격이 보이지 않음
- "오시는 곳", "자주 묻는 질문" 바로가기가 사라짐
- 세그먼트 컨트롤(기본 점검/정밀 점검/풀 정비)만 남아 내용 전환 가능

문제: 사이드바 콘텐츠가 메인 영역에 이미 존재하므로 기능 손실은 아니지만, 추가 메뉴 가격 비교를 위해 스크롤을 많이 해야 함.

### 5.2 하단 네비 + 콘텐츠 겹침

홈/진단/리포트에서 `.content`에 `padding-bottom: 100px`을 주어 하단 네비 뒤의 콘텐츠 가림을 방지. 그러나 요금제/결제/긴급 등 하단 네비가 없는 페이지는 이 패딩이 없어 일관성 부족.

### 5.3 터치 타겟

대부분의 버튼/행이 `padding: 12px` 이상으로 최소 44px 터치 타겟을 충족. 패키지 상세의 "+" 버튼(`32x32px`)은 44px 미만이므로 개선 필요.

## 7. 접근성 분석

| 항목 | 현재 상태 | 심각도 |
|------|----------|--------|
| 키보드 네비게이션 | `<button>` 사용으로 기본 focusable. 그러나 focus 스타일(`:focus-visible`) 미정의 | 높음 |
| Focus trap (모달) | 오버레이에 `role="dialog" aria-modal="true"` 있으나 focus trap 로직 없음 | 높음 |
| 스크린리더 | `aria-label`, `role="alert"`, `role="progressbar"` 일부 사용. 하지만 하단 네비에 `role="navigation"`, `aria-current` 없음 | 중간 |
| 색상 대비 | `--tx3: #4a4a4f` (#000 배경 위)는 WCAG AA 기준 미달 (약 2.5:1, 최소 4.5:1 필요) | 높음 |
| 동적 콘텐츠 알림 | 스캔 진행 상태 변경 시 `aria-live` 미사용 | 중간 |
| 건너뛰기 링크 | 없음 | 낮음 |

## 8. 개선 권고 5가지 (우선순위 순)

### 권고 1: 네비게이션 통합 및 "상담 예약" 승격 [긴급]

**문제**: 하단 네비가 3/9 페이지에만 존재. "상담 예약"이 고스트 버튼으로 눈에 띄지 않음.

**개선안**:
- 하단 네비를 공유 레이아웃 컴포넌트(`+layout.svelte`)로 추출
- 5탭 구성: 홈 | 진단 | 상담 | 리포트 | 요금제 ("상담"이 중앙 강조)
- 서브 페이지(리포트 상세, 패키지 상세, 결제)에서는 상단에 뒤로가기 + 하단 네비 유지
- 온보딩에서만 하단 네비 숨김

### 권고 2: 비즈니스 모델 단일화 -- 컨설팅 예약 우선 [긴급]

**문제**: 컨설팅 예약 모델(요금제)과 단품 장바구니 모델(리포트→결제)이 혼재.

**개선안**:
- 주 전환 동작을 "상담 예약"으로 통일
- 리포트 상세의 "추천 패키지 → 장바구니 → 결제" 흐름을 "이 문제를 해결하려면 → 상담 예약"으로 변경
- 312개 단품은 상담 과정에서 엔지니어가 제안하는 메뉴로 위치 재설정
- 장바구니/결제 페이지는 Phase 2로 미루거나 제거

### 권고 3: 온보딩 경량화 [높음]

**문제**: 첫 방문 시 강제 4단계 온보딩. 텍스트만 있고 인터랙션 없음.

**개선안**:
- 온보딩 강제 리다이렉트 제거
- 홈 페이지의 빈 상태(진단 기록 없을 때)를 온보딩 역할로 강화
- "이렇게 동작합니다" 3단계 설명을 홈 hero 아래 인라인 배치
- `/onboarding` 페이지는 제거하거나 선택적 접근으로 전환

### 권고 4: CSS 디자인 시스템 통합 [중간]

**문제**: 9개 페이지가 동일한 CSS 변수/스타일을 각자 선언. 바, 네비, 버튼이 복붙.

**개선안**:
- `app.css` 또는 `+layout.svelte`에 공통 CSS 변수 1회 선언
- `BottomNav.svelte`, `TopBar.svelte`, `EmptyState.svelte` 등 공유 컴포넌트 추출
- 버튼 스타일(`.cta-primary`, `.btn-outline`)을 전역 유틸리티 클래스로 이동

### 권고 5: 접근성 기초 확보 [중간]

**문제**: focus 스타일 없음, 모달 focus trap 없음, 색상 대비 미달.

**개선안**:
- `:focus-visible` 전역 스타일 추가 (최소 2px solid outline)
- `--tx3` 값을 `#7a7a7f` 이상으로 밝혀 WCAG AA 4.5:1 충족
- 모달/오버레이에 focus trap 라이브러리 또는 커스텀 로직 추가
- 하단 네비에 `<nav aria-label="주요 메뉴">`, 현재 탭에 `aria-current="page"` 추가
- 스캔 진행 상태에 `aria-live="polite"` 추가

## 10. 페이지 구조 재편 제안

### 현재 (9 페이지)


### 제안 (6 페이지, 핵심 플로우 단순화)

```
/                   홈 (온보딩 내용 인라인 포함, 빈 상태 강화)
/diagnose           진단 (URL 입력 + 스캔, 변경 없음)
/report             리포트 (목록 + [scanId] 상세를 같은 페이지 내 드릴다운)
/packages           요금제 (3개 플랜 중심, 단품은 상담 후 안내)
/contact            상담 예약 (incident 통합, 긴급/일반 모두 처리)
/packages/[id]      패키지 상세 (유지하되, 상담 CTA로 전환. 장바구니 제거)
```

**제거 대상**:
- `/onboarding` → 홈 빈 상태로 대체
- `/checkout` → Phase 2로 이관. 현재는 상담 예약 모델에 집중

**변경 사항**:
- `/incident` → `/contact`로 이름 변경. "긴급"이 아니라 "상담 예약"이 올바른 프레이밍
- 리포트 상세의 "추천 패키지" 섹션 CTA를 "장바구니 담기"에서 "상담 시 이 항목 논의하기"로 변경
- 하단 네비 5탭: 홈 | 진단 | **상담** | 리포트 | 요금제

## 부록: CSS 중복 현황

각 페이지에서 동일하게 선언된 CSS 변수 블록:

```
--black: #000000;
--s1: #060608;
--s2: #0c0c0e;
--s3: #141416;
--tx: #ffffff;
--tx2: #9a9a9f;
--tx3: #4a4a4f;
--brd: rgba(255,255,255,0.055);
```

이 블록이 **9개 파일 모두에 존재**. `.bar`, `.nav`, `.cta-primary`, `.btn-outline` 등의 스타일도 5개 이상의 파일에서 동일하게 반복됨. 공유 컴포넌트 + 전역 스타일로 추출하면 유지보수성과 일관성 동시 확보 가능.
